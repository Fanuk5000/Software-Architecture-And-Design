# Лабораторна 3, Варіант 9

## Завдання

В лабораторній роботі необхідно виокремити рівень доступу до даних, логіку роботи програми
(бізнес логіку) та інтерфейс користувача. Передбачити можливість додавання нового UI (наприклад,
замість використання консолі, з застосуванням також можна працювати і через WinForms).
Імплементувати інший UI не потрібно.
Головним критерієм якості є розділення на рівні, реалізація функціоналу та відповідність
базовим принципам проектування.
За необхідності можна додавати нові властивості, сутності та розширювати функціонал. Але ця
необхідність повинна бути обґрунтована.
Управління та відповідь застосування здійснюється через UI. Використання UI, наприклад, для
виведення повідомлень, потрібно винести за межі функціоналу предметної області (бізнес логіки).
Тобто, взаємодія з користувачем повинна бути за межами методів, що виконують певні дії.
Дані, з якими працює кожний рівень повинні бути представлені у виді сутностей (entities) на
відповідному рівні, а не використовувати безпосередньо сутності з рівня DAL.
Методи класів предметної області мають змінювати певним чином стан об’єкту, а не просто
виводити повідомлення. Виконання операцій повинно здійснюватися тільки при настанні визначених
умов.
Для демонстрації роботи програми необхідно підготувати набір вхідних даних.

- Квест-кімната :
Інформація по квест-кімнатах. По кожному квесту – ліміт на
кількість учасників. Реалізувати основний функціонал по пошуку
та бронюванню квесту. Замовити квест можна тільки у вільний
часовий проміжок. При наявності подарункового сертифікату
замовити квест можна безкоштовно.

## Відповіді на контрольні питання

**1. Що таке багаторівнева архітектура ПЗ та які переваги її використання?**
Багаторівнева (n-tier або layered) архітектура — це шаблон проектування, при якому застосунок логічно та/або фізично розділений на окремі рівні (шари), кожен з яких має свою чітку зону відповідальності.

* **Переваги:** Розділення відповідальності (Separation of Concerns), легкість підтримки та масштабування, можливість паралельної розробки різними командами, спрощене тестування (можна тестувати кожен рівень ізольовано), можливість заміни одного рівня без впливу на інші.

**2. Які існують шаблони проектування для організації шару доступу до сховищ даних?**
Основні шаблони для шару Data Access Layer (DAL):
* **Repository** (Репозиторій)
* **Unit of Work** (Одиниця роботи)
* **Data Mapper** (Відображення даних)
* **Active Record** (Активний запис)
* **Data Access Object / DAO** (Об'єкт доступу до даних)

**3. Поясніть ціль використання технології Entity Framework (EF) в архітектурі програмних систем.**
Entity Framework — це ORM (Object-Relational Mapping) фреймворк. Його головна ціль — дозволити розробникам працювати з реляційними базами даних за допомогою об'єктно-орієнтованого коду (класів та їх властивостей), а не писати "сирі" SQL-запити. EF бере на себе маршрутизацію запитів, мапінг даних з таблиць у об'єкти та відстеження змін (change tracking).

**4. Поясніть сутність шаблону Repository.**
Репозиторій — це шар абстракції між логікою предметної області та рівнем доступу до даних. Він працює як колекція об'єктів у пам'яті: ти можеш додати сутність, видалити або знайти її. Сутність цього шаблону — приховати складність роботи з базою даних від бізнес-логіки. Бізнес-логіка не знає, чи це база даних, чи текстовий файл, вона просто викликає методи на кшталт `GetById()` або `Add()`.

**5. В чому полягає особливість використання generic репозиторію.**
Особливість Generic (узагальненого) репозиторію в тому, що він використовує типи (generics у C# або Python) для створення одного універсального класу, який може обробляти базові CRUD (Create, Read, Update, Delete) операції для *будь-якої* сутності (наприклад, `Repository<User>`, `Repository<Order>`). Це дозволяє уникнути дублювання коду, оскільки не потрібно писати окремий ідентичний репозиторій для кожної таблиці бази даних.

**6. Поясніть сутність шаблону Unit of Work.**
Unit of Work (Одиниця роботи) відповідає за управління транзакціями бази даних. Він відстежує всі зміни (додавання, оновлення, видалення), які виконуються різними репозиторіями в рамках однієї бізнес-операції, і гарантує, що ці зміни будуть збережені в базі даних одночасно (однією транзакцією через `commit`) або повністю скасовані (через `rollback`) у разі помилки. 

**7. Як пов’язані EF та шаблони Repository та Unit of Work?**
Entity Framework вже сам по собі реалізує ці шаблони:
* Клас `DbContext` в EF — це по суті реалізація **Unit of Work** (він відстежує зміни і має метод `SaveChanges()`).
* Клас `DbSet<T>` — це реалізація **Repository** для конкретної сутності.
* **Чому їх тоді пишуть окремо?** Ми створюємо власні обгортки (кастомні Repository та UoW) навколо EF, щоб повністю абстрагувати бізнес-логіку від самого Entity Framework. Це дозволяє в майбутньому легко замінити EF на іншу ORM без переписування бізнес-логіки.

**8. Поясніть сутність процесу mapping.**
Mapping (відображення/мапінг) — це процес перетворення об'єкта одного типу на об'єкт іншого типу. В архітектурі це найчастіше копіювання даних з об'єкта бази даних (Entity) в об'єкт для передачі даних (DTO) або модель для відображення (ViewModel). Це робиться для того, щоб не "світити" структуру бази даних на верхніх рівнях та не передавати зайві або конфіденційні дані (наприклад, не передавати хеш пароля користувача на UI).

**9. Наведіть шари та їх призначення в багаторівневій архітектурі ПЗ.**
Традиційна 3-рівнева архітектура включає:
* **Presentation Layer / UI (Рівень представлення):** Відповідає за взаємодію з користувачем, приймання запитів та відображення даних (наприклад, Web API, WPF, Console).
* **Business Logic Layer / BLL (Рівень бізнес-логіки):** Серце програми. Містить правила, розрахунки та логіку предметної області. Обробляє дані перед збереженням або відображенням.
* **Data Access Layer / DAL (Рівень доступу до даних):** Відповідає виключно за збереження та отримання даних з БД, файлів чи зовнішніх API.

**10. Опишіть властивості шарів багаторівневої архітектури ПЗ.**
* **Висока зв'язність (High Cohesion):** Всередині одного шару знаходяться класи, що виконують строго пов'язані між собою задачі.
* **Слабке зачеплення (Loose Coupling):** Шари мають мінімальну залежність один від одного. Зазвичай це досягається через використання інтерфейсів (Dependency Injection).
* **Односпрямованість:** Верхні рівні можуть викликати нижні (UI викликає BLL, BLL викликає DAL), але нижні рівні нічого не знають про верхні (DAL не може викликати методи UI).

**11. В чому полягає принцип ізоляції рівнів у багаторівневій архітектурі.**
Принцип полягає в тому, що зміни всередині одного рівня не повинні призводити до каскадних змін в інших. Рівні спілкуються виключно через чітко визначені "контракти" (інтерфейси) та передають абстрактні DTO, а не конкретні реалізації. Наприклад, завдяки ізоляції ти можеш повністю замінити базу даних (DAL), і рівень бізнес-логіки цього навіть не помітить.

**12. Яка користь від створення та використання сутностей (entities) на кожному рівні багаторівневої архітектури?**
Кожен рівень працює зі своїм специфічним набором вимог. 
* DAL має об'єкти (Entities), обвішані атрибутами для бази даних (зовнішні ключі, ліміти довжини рядка).
* BLL потребує чистих об'єктів для розрахунків.
* UI потребує об'єктів (ViewModels/DTO) з повідомленнями про помилки валідації для користувача (наприклад, "Пароль надто короткий").
Використання окремих моделей для кожного рівня гарантує, що архітектурні деталі одного рівня (наприклад, налаштування таблиць БД) не будуть "забруднювати" інші рівні.
